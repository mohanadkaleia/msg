## 2. Design decisions (ruled)

### 2.1 Schema generation â€” GENERATED, not hand-written

**Rule: generated by a committed script + frozen by a test.** Same discipline as `vectors.json`.

- `server/tests/generate_schemas.py` imports `Envelope` and `MessageCreatedV1`, calls
  `model_json_schema()`, wraps each with `"$schema": "https://json-schema.org/draft/2020-12/schema"`,
  a stable `"$id"` (e.g. `https://msg.dev/schemas/envelope.schema.json`), and a `"title"`, then writes
  `docs/schemas/envelope.schema.json` and `docs/schemas/message.created.v1.schema.json` with a **deterministic
  serialization**: `json.dumps(doc, indent=2, sort_keys=True, ensure_ascii=True) + "\n"`.
- Path resolution mirrors `generate_vectors.py`: `Path(__file__).resolve().parents[2] / "docs" / "schemas"`
  (`server/tests/` â†’ parents[2] = repo root). Create `docs/schemas/` if absent.
- `server/tests/test_schemas.py` regenerates in memory and asserts **byte-equality** to the committed files
  (no SHA constant needed â€” a direct string compare is self-contained, unlike vectors where the file is data
  consumed elsewhere). On mismatch the message says: regenerate via
  `uv run python server/tests/generate_schemas.py`.
- **Why generated:** hand-written JSON Schema would immediately drift from the Pydantic models that are the
  actual runtime contract; generation makes the models the single source of truth and the freeze test makes the
  committed artifact a reviewed, diffable output. `model_json_schema()` output is pinned to the locked pydantic
  version (`uv.lock`); document in the script docstring that a pydantic upgrade that changes schema output is a
  deliberate regenerate-and-review event, exactly like a vector change.
- **Scope of "envelope" schema:** the full `Envelope` model (body + `event_hash` + `signature` + `server`),
  because that is the stored/served shape Â§2.2 points plugin authors at. `Body`/`ServerMetadata` appear as
  `$defs`. `payload` renders as an open object (`extra="allow"` + `dict[str, Any]`) â€” correct, since the
  envelope deliberately keeps payload opaque (D9); the typed payload contract lives in the separate
  `message.created.v1` schema.

### 2.2 Dogfood-scale test â€” CI-permanent, deterministic, <60s

**Rule: `cli/tests/test_dogfood_scale.py`, kept in the normal CI pytest run (not skipped), sized to a
deterministic ~5k events / 8 streams, target wall-clock < 45s (60s hard ceiling).** It doubles as a perf canary.

- Build via **library calls** (same helpers as `test_equivalence_gate.py`: `init_workspace`, `Workspace.open`,
  `resolve_or_create_stream`, `append_event` with a `build_message_created_body` closure) â€” never subprocess
  per send.
- **Deterministic, not hypothesis:** use a fixed `random.Random(seed)` (or plain round-robin) to distribute
  5000 `message.created` across 8 streams with varied text (include unicode + a few unknown-type injections so
  the D9 skip path is exercised at scale). A fixed count keeps runtime bounded and the perf-canary signal
  meaningful.
- **One `project()` call over the whole log** (the incremental cursor path from 0â†’head), then
  `rebuild_projection(ws)`, then compare `dump_messages` byte-for-byte (the equivalence assertion at scale).
  Then `verify.verify_workspace(root).exit_code == 0` â€” this single test **also satisfies the checklist's
  "verify green on a dogfood-sized synthetic workspace" item**.
- **Runtime note for the implementer:** `append_event` fsyncs per event (`append.py` L211) â€” on Linux CI this
  is sub-millisecond; 5k events + one project + one rebuild + one verify should land well under budget. Measure
  on first run; if > 45s, trim to ~3k and record the actual number in the module docstring. Optional
  `@pytest.mark.slow` (registered in root `pyproject.toml`) so local devs can `-m "not slow"`, but **CI runs the
  full suite so it must not be deselected there** â€” do not add a skip.

## 6. Risks / open questions

- **`model_json_schema()` version sensitivity.** Output can shift across pydantic minor versions. Mitigation:
  the freeze test pins it to the locked version; a bump becomes a deliberate regenerate step. Documented in the
  generator docstring. (Low risk â€” pydantic is pinned in `uv.lock`.)
- **Dogfold runtime creep as a perf canary.** 5k fsyncing appends could exceed budget on a slow runner.
  Mitigation: measure on first CI run, trim count and record it; hard ceiling 60s. This is a feature (canary),
  not just a risk.
- **Duplicate-key vector portability caveat.** The vector pins *last-wins*, which CPython `json` and V8
  `JSON.parse` both honor; the desc must state consumers MUST use a last-wins parser (already in the desc text).
  A pathological TS parser that throws on duplicate keys is out of scope â€” the vector's job is to make the
  requirement explicit, not to accommodate every parser.

  D-table/Â§2 revision and must be its own decision â€” **not** absorbed into this gate PR. Escalate to tech-lead.
